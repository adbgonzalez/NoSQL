# Laboratorio de Data Prepper
**Data Prepper** é unha ferramenta de inxesta integrada no ecosistema **OpenSearch**, alternativa a **Logstash** no ecosistema **ELK**. Para máis información, consultar o documento correspondente.

Neste laboratorio imos configurar un stack de docker para executar algúns exemplos básicos de inxesta de datos con **Data Prepper**. Para iso imos realizar os seguintes pasos:
1. Preparar e configurar o entorno docker.
2. Crear e validar as pipelines de Data Prepper.
3. Executar os produtores de datos.
4. Verificar a indexación en OpenSearch.
5. Visualizar os datos en OpenSearch Dashboards.
6. Realizar pequenas modificacións e probas.


## 1. Preparar e configurar o entorno docker
Imos partir do `docker-compose.yml` realizado en sesións previas que contén un contedor de `opensearch-dashboards` e outro de `opensearch`. Imos engadir:
- Un contedor de `data-prepper` empregando a imaxe oficial `opensearchproject/data-prepper:2`. Este contedor conterá dous volumes, un para o directorio de `pipelines` e outro para o ficheiro de configuración `data-prepper-config.yaml`.
```yaml
  data-prepper:
    image: opensearchproject/data-prepper:2
    container_name: data-prepper
    depends_on:
      - opensearch
    ports:
      - "2021:2021"
      - "4900:4900"
    volumes:
      - ./pipelines:/usr/share/data-prepper/pipelines
      - ./data-prepper-config.yaml:/usr/share/data-prepper/config/data-prepper-config.yaml
    networks:
      - opensearch-net
```
- Un contedor de `producer` baseado nunha imaxe `python:3.11-slim` que será o encargado de executar os scripts de python que producián os datos a inxestar en *OpenSearch*.
```yaml
  producer:
    image: python:3.11-slim
    container_name: producer
    depends_on:
      - data-prepper
    working_dir: /app
    volumes:
      - ./producers:/app
      - producer-pip:/pip
    environment:
      - PIP_CACHE_DIR=/pip/cache
      - PYTHONPATH=/pip/site-packages
    command: >
      sh -c "pip install --no-cache-dir --target=/pip/site-packages -r /app/requirements.txt &&
             sleep infinity"
    networks:
      - opensearch-net
```
O ficheiro `docker-compose.yml` completo sería parecido a isto:
```yaml
services:
  opensearch:
    image: opensearchproject/opensearch:2
    container_name: opensearch
    environment:
      - cluster.name=opensearch-cluster
      - node.name=opensearch-node
      - discovery.type=single-node
      - bootstrap.memory_lock=true
      - OPENSEARCH_JAVA_OPTS=-Xms1g -Xmx1g
      - OPENSEARCH_INITIAL_ADMIN_PASSWORD=${OPENSEARCH_INITIAL_ADMIN_PASSWORD}
    ulimits:
      memlock:
        soft: -1
        hard: -1
    volumes:
      - opensearch-data:/usr/share/opensearch/data
    ports:
      - "9200:9200"
      - "9600:9600"
    networks:
      - opensearch-net

  opensearch-dashboards:
    image: opensearchproject/opensearch-dashboards:2
    container_name: opensearch-dashboards
    ports:
      - "5601:5601"
    environment:
      - OPENSEARCH_HOSTS=["https://opensearch:9200"]
    depends_on:
      - opensearch
    networks:
      - opensearch-net

  data-prepper:
    image: opensearchproject/data-prepper:2
    container_name: data-prepper
    depends_on:
      - opensearch
    ports:
      - "2021:2021"
      - "4900:4900"
    volumes:
      - ./pipelines:/usr/share/data-prepper/pipelines
      - ./data-prepper-config.yaml:/usr/share/data-prepper/config/data-prepper-config.yaml
    networks:
      - opensearch-net

  producer:
    image: python:3.11-slim
    container_name: producer
    depends_on:
      - data-prepper
    working_dir: /app
    volumes:
      - ./producers:/app
      - producer-pip:/pip
    environment:
      - PIP_CACHE_DIR=/pip/cache
      - PYTHONPATH=/pip/site-packages
    command: >
      sh -c "pip install --no-cache-dir --target=/pip/site-packages -r /app/requirements.txt &&
             sleep infinity"
    networks:
      - opensearch-net

volumes:
  opensearch-data:
  producer-pip:
networks:
  opensearch-net:
```
> Nota: non é necesario publicar no host o porto 2022, xa que os produtores executan dentro do contedor `producer` e comunícanse con `data-prepper` a través da rede interna de Docker.

## 2. Creación manual dos índices con mapping explícito

Antes de iniciar a inxesta de datos, créanse manualmente os índices en OpenSearch definindo explicitamente o *mapping*. 

Isto evita a inferencia automática de tipos e garante:

- coherencia no modelo de datos
- agregacións correctas
- filtros eficientes
- estabilidade a longo prazo

> Se non se define o mapping, OpenSearch decidirá o tipo segundo o primeiro documento recibido.

---

### Índice de logs

No menú **Dev Tools → Console** de OpenSearch Dashboards, executar:

```http
PUT logs_http
{
  "mappings": {
    "properties": {
      "timestamp": { "type": "date" },
      "servizo":   { "type": "keyword" },
      "nivel":     { "type": "keyword" },
      "latency_ms":{ "type": "integer" }
    }
  }
}
```

Decisións de modelado:

- `timestamp` → `date` (permite filtros temporais e `date_histogram`)
- `servizo` → `keyword` (campo categórico)
- `nivel` → `keyword` (info, warn, error)
- `latency_ms` → `integer` (permite agregacións numéricas como `avg`)

---

### Índice de bicicletas

```http
PUT citybikes_http
{
  "mappings": {
    "properties": {
      "timestamp":    { "type": "date" },
      "orixe":        { "type": "keyword" },
      "network_id":   { "type": "keyword" },
      "network_name": { "type": "keyword" },
      "station_id":   { "type": "keyword" },
      "station_name": { "type": "text" },
      "free_bikes":   { "type": "integer" },
      "empty_slots":  { "type": "integer" },
      "latitude":     { "type": "float" },
      "longitude":    { "type": "float" },
      "last_updated": { "type": "date" }
    }
  }
}
```

Decisións de modelado:

- Campos identificadores → `keyword`
- Campos numéricos → `integer` ou `float`
- Datas → `date`
- Nome da estación → `text` (permite busca textual)

---

### Verificación do mapping

Unha vez creados os índices, pódese comprobar o mapping:

```http
GET logs_http/_mapping
```

```http
GET citybikes_http/_mapping
```

Se o mapping é correcto, xa se pode iniciar a inxesta executando os produtores.

---

### Nota importante

Se os produtores xa enviaron datos e o índice foi creado automaticamente, será necesario:

1. Eliminar o índice:
   ```http
   DELETE logs_http
   ```
2. Recrealo co mapping correcto.
3. Reiniciar os produtores.

En contornos reais, este control realízase mediante **index templates**, pero neste laboratorio créanse manualmente para comprender o proceso.

## 3. Crear e validar as pipelines

No directorio `pipelines/` temos dúas pipelines: unha para logs simulados e outra para datos de bicicletas.

### 3.1 Pipeline de logs

Ficheiro: `logs-to-opensearch.yml`

```yaml
logs-to-opensearch:
  source:
    http:
      path: "/events/logs"
      port: 2022

  sink:
    - opensearch:
        hosts: [ "https://opensearch:9200" ]
        insecure: true
        username: "admin"
        password: "Opensearch#2025"
        index: "logs_http"
```

Esta pipeline:

- Expón un endpoint HTTP en `2022/events/logs`
- Recibe eventos en formato JSON
- Indexa os datos no índice `logs_http`

> É importante que o porto coincida co definido no script `log_producer_http.py`.

---

### 3.2 Pipeline de bicicletas

Ficheiro: `bikes-to-opensearch.yml`

```yaml
bikes-to-opensearch:
  source:
    http:
      path: "/events/bikes"
      port: 2021

  sink:
    - opensearch:
        hosts: [ "https://opensearch:9200" ]
        insecure: true
        username: "admin"
        password: "Opensearch#2025"
        index: "citybikes_http"
```

Esta pipeline:

- Expón o endpoint `2021/events/bikes`
- Recibe datos da API pública de citybik.es
- Indexa no índice `citybikes_http`

## 4. Arrincar o stack

Desde o directorio do proxecto:

```bash
docker compose up -d
```

Comprobar que todos os contedores están activos:

```bash
docker ps
```

Deberían aparecer:

- opensearch
- opensearch-dashboards
- data-prepper
- producer

> Podemos asegurarnos de que está todo listo se esperamos a que a interfaz de opensearch-dashboards está activa.
Comprobar logs de Data Prepper:

```bash
docker logs -f data-prepper
```

Se as pipelines cargan correctamente, aparecerán mensaxes indicando que os endpoints HTTP están activos.

## 5. Executar os produtores

Acceder ao contedor producer:

```bash
docker exec -it producer sh
```

### 5.1 Executar produtor de logs

```bash
python log_producer_http.py
```

Cada 2 segundos enviará un evento como:

```json
{
  "timestamp": "...",
  "servizo": "api",
  "nivel": "info",
  "latency_ms": 345
}
```

---

### 5.2 Executar produtor de bicicletas

Nun segundo terminal:

```bash
docker exec -it producer sh
python bike_producer_http.py
```

Cada 60 segundos consultará a API de citybik.es e enviará os datos a Data Prepper.

## 8. Experimentos propostos

### 8.1 Engadir un processor

Modificar a pipeline de logs para engadir un campo:

```yaml
processor:
  - add_entries:
      entries:
        - key: "entorno"
          value: "laboratorio"
```

Reiniciar Data Prepper:

```bash
docker compose restart data-prepper
```

Comprobar que o novo campo aparece nos documentos indexados.

---

### 8.2 Filtrar só erros

Engadir un processor `drop_events` para descartar eventos que non sexan erro.

## Tarefa — Inxesta de “access logs” cun producer novo

### Obxectivo
Configurar unha inxesta completa producer → Data Prepper → OpenSearch a partir dun novo produtor en Python (fornecido), creando:
1. o pipeline de Data Prepper (YAML)
2. o índice en OpenSearch con mapping explícito
3. verificación de que os datos quedaron indexados

---

## 1. Producer 

Crear o ficheiro `producers/access_producer_http.py` co seguinte contido:

```python
import time
import random
import requests
from datetime import datetime, timezone

DATAPREPPER_URL = "http://data-prepper:2023/events/access"
POLL_SECONDS = 2
TIMEOUT = 5

SERVICES = ["web", "api", "auth"]
METHODS = ["GET", "POST", "PUT", "DELETE"]
ENDPOINTS = ["/", "/login", "/logout", "/api/v1/items", "/api/v1/orders", "/health"]
STATUS_CODES = [200, 201, 204, 400, 401, 403, 404, 500, 502, 503]
USER_AGENTS = [
    "Mozilla/5.0",
    "curl/8.0",
    "PostmanRuntime/7.36",
    "Python-requests/2.31",
]
IPS = ["10.0.0.10", "10.0.0.11", "10.0.0.12", "192.168.1.20", "172.16.0.5"]

def now_utc_iso():
    return datetime.now(timezone.utc).isoformat().replace("+00:00", "Z")

print(f"[access] enviando a {DATAPREPPER_URL}")

while True:
    servizo = random.choice(SERVICES)
    method = random.choice(METHODS)
    endpoint = random.choice(ENDPOINTS)
    status = random.choice(STATUS_CODES)

    latency_ms = round(random.uniform(5, 1200), 2)
    bytes_out = random.randint(200, 20000)

    evento = {
        "timestamp": now_utc_iso(),
        "servizo": servizo,
        "client_ip": random.choice(IPS),
        "method": method,
        "endpoint": endpoint,
        "status": status,
        "latency_ms": latency_ms,
        "bytes_out": bytes_out,
        "user_agent": random.choice(USER_AGENTS),
        "mensaxe": f'{method} {endpoint} -> {status} ({latency_ms} ms)'
    }

    try:
        r = requests.post(DATAPREPPER_URL, json=[evento], timeout=TIMEOUT)
        print("status:", r.status_code, "| enviado:", evento["mensaxe"])
    except Exception as e:
        print("[access] erro:", repr(e))

    time.sleep(POLL_SECONDS)
```

Requisito: en `producers/requirements.txt` debe estar `requests`.

## 2. Tarefas do alumnado

### 2.1 Crear o índice con mapping explícito (obrigatorio)
Crear o índice `access_http` en Dev Tools → Console, definindo tipos correctos para:
- `timestamp` como `date`
- `servizo`, `client_ip`, `method`, `endpoint` como `keyword`
- `status` como `integer`
- `latency_ms` como `float`
- `bytes_out` como `integer`
- `user_agent` como `keyword` (ou `text` se se quere busca full-text)
- `mensaxe` como `text`

Nota: o índice debe existir antes de executar o producer, para evitar inferencia automática.

### 2.2 Crear un pipeline novo en Data Prepper
Crear o ficheiro `pipelines/access-to-opensearch.yml` que:
- teña unha `source` de tipo `http`
- escoite no porto `2023`
- use o path `/events/access`
- teña un `sink` cara OpenSearch que indexe en `access_http`

Non é necesario publicar o porto 2023 no host se non se fan probas fóra de Docker.

### 2.3 Executar o producer
Desde o contedor `producer`:

```bash
python access_producer_http.py
```

## 3. Verificación 

### 3.1 Ver que o índice existe e crece

```http
GET _cat/indices/access_http?v
```

ou:

```http
GET access_http/_count
```

### 3.2 Consultar algúns documentos

```http
GET access_http/_search
{
  "size": 5,
  "sort": [
    { "timestamp": "desc" }
  ]
}
```

### 3.3 Proba de agregación (para validar o mapping)

```http
GET access_http/_search
{
  "size": 0,
  "aggs": {
    "por_endpoint": {
      "terms": { "field": "endpoint" },
      "aggs": {
        "latencia_media": { "avg": { "field": "latency_ms" } }
      }
    }
  }
}
```

## 4. Entregables e guía de corrección

### 4.1. Entregables
- Saída de:
  - `GET access_http/_mapping`
  - `GET access_http/_count`
  - unha consulta `_search` con resultados
  - a agregación de latencia media


### 4.2 Guía de corrección


| Nº | Descrición | Evidencia na que se atopa | Puntuación |
|----|------------|---------------------------|------------|
| 1 | Índice `access_http` creado manualmente antes da inxesta | Captura do `PUT access_http` en Dev Tools | 1 punto |
| 2 | `timestamp` definido como `date` | Saída de `GET access_http/_mapping` | 1 punto |
| 3 | `status` definido como `integer` | Saída de `GET access_http/_mapping` | 1 punto |
| 4 | `latency_ms` definido como `float` | Saída de `GET access_http/_mapping` | 1 punto |
| 5 | Campos categóricos (`servizo`, `endpoint`, etc.) definidos como `keyword` | Saída de `GET access_http/_mapping` | 1 punto |
| 6 | Pipeline HTTP configurado con porto `2023` e path `/events/access` | Ficheiro `pipelines/access-to-opensearch.yml` | 1 punto |
| 7 | Sink correctamente configurado cara ao índice `access_http` | Ficheiro `pipelines/access-to-opensearch.yml` | 1 punto |
| 8 | Existen documentos indexados (count > 0) | Saída de `GET access_http/_count` | 1 punto |
| 9 | Consulta `_search` devolve documentos recentes ordenados por `timestamp` | Captura ou saída de `GET access_http/_search` | 1 punto |
| 10 | Agregación de latencia media por `endpoint` funciona correctamente | Saída da consulta con `avg(latency_ms)` | 1 punto |

**Total: 10 puntos**
